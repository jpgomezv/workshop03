Workshop Goal

Design and implement a streaming ETL pipeline capable of generating real-time predictions using Apache Kafka and a pre-trained machine learning model.

  • Integrate 5 heterogeneous CSV datasets
  • Clean, harmonize, engineer features
  • Train and serialize a regression model
  • Stream raw data through Kafka producer
  • Validate events & run real-time inference
  • Store in a star schema, visualize with a BI tool

happiness_rank -0.99 No — leakage
country — No — 170 cats
continent — Optional (excluded)

Why 6 numeric features? They match the Kafka JSON format exactly. The consumer extracts them directly — no lookup, no extra complexity.

Scaling: StandardScaler applied post-split. Features are already in 0–2 range but scaling ensures consistency.

Continent experiment: Adding continent dummies improves R² by only +0.04. Not worth the consumer complexity for a streaming pipeline.

Part B · Step 5 — Kafka Producer

Producer — Streaming Raw Data

What it does

  • Reads 5 raw CSV files from data/raw/
  • Maps 22 column names → 9 unified fields
  • No cleaning — raw values preserved
  • Converts NaN → null (valid JSON)
  • Streams each row one-by-one to Kafka

JSON Format

{"country":"Colombia","year":2019,
"gdp":1.2,"family":0.8,"health":0.9,
"freedom":0.6,"generosity":0.3,
"corruption":0.1,
"actual_happiness_score":6.2}

Key details

Uses kafka-python-ng library · Topic: happiness-predictions · 0.5s delay between messages · 782 total events · Country+year as message key for partitioning

Why raw data?

Part A (training) used cleaned data. Part B (streaming) starts from scratch — this validates the consumer's ability to handle messy, unprocessed data in real time.

Includes the UAE 2018 record with corruption=null — tests the validation pipeline

Part B · Step 6 — Kafka Consumer

Consumer — Processing Flow

  1. 1 Receive event from Kafka topic
  2. 2 Store raw event immediately in raw_happiness_events (status=RECEIVED)
  3. 3 Populate dim_raw_event dimension
  4. 4 Validate event schema & values
  5. 5 If invalid → update status, skip prediction, continue
  6. 6 Extract features in model order: [gdp, family, health, freedom, generosity, corruption]
  7. 7 Load model.pkl → model.predict()
  8. 8 UPSERT dim_country and dim_date
  9. 9 INSERT into fact_predictions with raw_event_id link
  10. 0 Mark as VALID

Key: raw events stored before validation — supports traceability, auditing, and future reprocessing

Part B · Step 6 — Consumer

Validation & Error Handling

Validation Checks

  • Missing required fields → INVALID_SCHEMA
  • Wrong data types (e.g., "year" as string) → INVALID_SCHEMA
  • Null/NaN in numeric features → INVALID_VALUES
  • Missing features required by the model → INVALID_VALUES

Processing Statuses

RECEIVED Initial state, before validation
VALID All checks passed · Prediction stored
INVALID_SCHEMA Missing field or wrong type
INVALID_VALUES Null in a required field
PREDICTION_ERROR Model prediction failed

Invalid records are stored in the raw table with their status, skipped from prediction, and never crash the pipeline.

781 VALID predictions stored

1 INVALID_VALUES — UAE 2018 (corruption=null)

Part C · Step 7 — Database Design

Star Schema Data Model

Table Type Rows Purpose
raw_happiness_events Raw Table 782 Original Kafka messages + processing_status + raw_payload (JSONB)
dim_raw_event Dimension 782 Links fact to raw event metadata (received_at, status)
dim_country Dimension 170 Country name → country_id lookup
dim_date Dimension 5 Year → date_id lookup
fact_predictions Fact Table 781 actual_score, predicted_score, prediction_error, prediction_timestamp

Entity Relationship: fact_predictions.raw_event_id → raw_happiness_events · fact_predictions.country_id → dim_country · fact_predictions.date_id → dim_date

Each prediction can be traced back to the exact raw event that produced it

Key Decisions

Design Tradeoffs & Takeaways

6 features over continent encoding. Adding continent improved R² by only +0.04, but requires a country→continent lookup in the consumer. Pipeline simplicity won.

Raw events stored before validation. This supports traceability, auditing, and reprocessing if prediction rules change. Invalid events are preserved, not discarded.

Star schema for analytical queries. Three dimensions (country, date, raw_event) feed a single fact table. This is the standard pattern for BI and analytical workloads.

Graceful error handling. The consumer never crashes on bad data. DB connection drops trigger automatic reconnection. Invalid events are logged and skipped.
